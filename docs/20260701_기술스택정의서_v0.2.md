# 기술스택 정의서 — 나래 AI 프로토타입 (Claude Code용)

> **버전** v0.2 · **작성일** 2026-07-01 · **성격** 디자인·인터랙션 프로토타입(백엔드 없음, mock)
> **읽는 주체** Claude Code. 본서 + 3종 정의서(기능·화면·입출력) **v0.8** 을 **진실원천(SoT)** 으로 읽고 구현 계획을 수립한다.
> **v0.2 변경** 0.7ver 화면설계서·결정 D2~D6 확정(v0.8) 반영 — **화면ID 재편·GNB 4개·홈 인피니티 캔버스·라우팅 맵 전면 갱신**. 스택에 노드 캔버스(React Flow) 추가.
> **출처** 스택은 `docs/근거/기술 초기 정리.xlsx > 기술 스펙 정리` 시트의 프론트엔드 스택과 정렬(백엔드·실API 제외).

---

## 1. 원칙

1. **프로토타입이다.** 백엔드·실 DB·실인증 없음. 모든 데이터는 `src/mock`의 시드. 비동기는 setTimeout으로 흉내.
2. **정의서가 SoT.** 화면·기능·데이터·인터랙션은 3종 xlsx 정의서 **v0.8** 을 따른다. 임의 추측 금지 — 근거 없으면 `// TODO: 기획확인` 주석, 정의서 `[확인필요]` 항목은 임시값+주석.
3. **화면ID 기준.** 모든 라우트·페이지 컴포넌트·mock·테스트는 정의서의 `SCR-*` **신 ID(v0.8)** 로 매핑·추적한다.
4. **KRDS 준수.** krds-react 기본 컴포넌트 우선. 없는 것(간트·칸반·캘린더·노드 캔버스)만 커스텀.
5. **토큰 우선.** 색·간격·타이포는 하드코딩 금지, `design/tokens.css` 변수만 사용.

```

├─ docs/                 # 본 정의서 4종(기능·화면·입출력·스택) v0.8/v0.2
├─ design/
│   ├─ tokens.css        # 디자인 토큰(CSS 변수)
│   └─ mockups/          # 화면별 HTML 목업(Claude Design 산출)
├─ src/
│   ├─ components/       # 공통: Layout/Gnb/TopBar/AiSearchBar/Bell/Card/Table/FilterChip/Tabs/Drawer/Badge/Uploader/Gantt/Calendar/Kanban/NodeCanvas/ToolPalette
│   ├─ pages/            # 화면: SCR-* 1:1 (영역 폴더: home/org/work/meeting/reg/bi/settings/admin/common) — §7 참조
│   ├─ scenarios/        # 시나리오 흐름(예: 과제 등록→AI 분석→업무 상세/편집)
│   ├─ mock/             # 엔터티별 시드 JSON (아래 목록, 입출력정의서 3_스키마와 동일 키)
```

## 4. 디자인 토큰 규약 (`design/tokens.css`)

- 형태: `:root { --color-primary: #2456E6; --space-2: 8px; ... }` CSS 변수.
- 그룹: color(primary/ok/warn/risk/ink/line/bg/surface) · space · radius · shadow · font(family/size/weight).
- 컴포넌트는 토큰만 참조. **하드코딩 색/픽셀 금지**(코드리뷰 차단 항목).
- KRDS 톤 + 분석 사이트(네이비 헤더·정부 블루) 반영. 토큰 초안은 Claude Design 1차 산출물.

## 5. 컴포넌트 규약

- 공통 컴포넌트 먼저 → 화면은 조립. props 최소·기본값 제공.
- krds-react에 있으면 래핑해서 사용(직접 스타일 덮어쓰기 금지, 토큰으로).
- 커스텀(간트·칸반·캘린더·노드 캔버스)은 `components/`에 독립, mock 데이터 주입.
- 화면 컴포넌트 = `pages/<area>/<Screen>.jsx`, 파일 상단 주석에 대응 `SCR-*` ID 기재.
- **Drawer 패턴**: 업무 상세(WF-02)·홈 BI/도구 Drawer는 라우트 오버레이(배경 유지). URL은 `?drawer=` 쿼리 또는 중첩 라우트로 표현.

## 6. mock 데이터·상태 규약

- `src/mock/<entity>.js` 가 단일 시드 소스(입출력정의서 v0.8 `3_스키마`와 동일 키).
- 비동기 흉내: `await sleep(ms)` 후 mock 반환(로딩 스피너 노출).
- zustand: `useRoleStore`(역할·조회범위), `useFilterStore`(화면별 필터), `useUiStore`(모달·드로어·토스트), `useCanvasStore`(홈/조직도뷰 노드·엣지·전개 경로·도구 부착).
- 역할 전환 시 화면 데이터 범위만 mock에서 필터(권한 분기 흉내). 권한 매트릭스는 기능상세정의서 `3_권한매트릭스`.

## 9. Claude Code 작업 규칙 (CLAUDE.md에 반영)

- 작업 단위: **화면 1개씩**. 공통 컴포넌트·토큰·노드 캔버스 먼저.
- 새 작업/버전업 시 **읽는 순서**: ① 본 스택 정의서 → ② 화면상세정의서 v0.8(해당 화면 + 인터랙션) → ③ 기능상세정의서 v0.8(해당 화면 기능) → ④ 입출력데이터정의서 v0.8(해당 화면 입출력·스키마). (specs는 갱신 보류 → 참조하지 않음)
- 금지: 백엔드·실API·실인증 구현, 토큰 외 하드코딩, 정의서에 없는 임의 데이터(필요 시 TODO).
- 산출 후: 해당 화면의 인터랙션 정의 행(화면상세 `3_버튼별인터랙션`)과 1:1 동작 확인(스펙↔구현 셀프 체크).
- 정의서 `[확인필요]` 표기 항목은 임시 구현 + `// TODO: 기획확인(SCR-xx)` 주석으로 남긴다.

## 10. 요구사항 변경(버전업) 흐름

1. 기획 변경 → 영향 화면의 정의서 행 수정 + 버전·일자 갱신 + **버전관리대장 §5·§6 기록**(무엇이/왜/영향 화면).
2. UI 변화 있으면 `design/`(토큰·목업) 갱신.
3. 화면ID 체계 등 **구조 변경은 전 정의서 일괄 반영**(부분 수정 금지).
4. Claude Code가 해당 화면/컴포넌트만 수정 → Vercel 재배포. 변경은 **화면 단위로 격리**.
