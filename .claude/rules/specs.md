## 스펙 기반 구현 규칙

### 구현 시작 전
- 구현 대상 화면의 `specs/SCR-*.md`를 먼저 읽는다.
- 인터랙션 표의 INT-ID 목록을 추출해 todo 리스트로 만든다.
- 플랜 모드로 구현 계획과 검증 전략을 먼저 제시한다.

### 구현 중
- INT 1개 = 1커밋 단위: `feat: INT-<SCR>-<nn> <요약>`
- 각 INT의 수용기준(Given-When-Then)을 코드에 반영한다.
- 트리거(클릭/입력/드래그 등) → 동작 → 결과/피드백 흐름을 정확히 구현.
- 전제조건·예외(비활성/에러)도 스펙에 명시된 대로 처리.

### mock 데이터
- 키 이름은 입출력정의서 스키마와 정확히 일치시킨다.
- 임의 필드 추가 금지. 부족하면 `[CONFIRM: 필드명 확인필요]` 남기고 질문.
- `src/mock/<entity>.js`가 단일 시드 소스. 비동기 흉내: `await sleep(ms)` 후 반환.

### 상태 5종 필수 구현
모든 화면에 아래 5가지 상태를 구현한다:
1. **초기**: 화면 진입 시 기본 표시
2. **로딩**: 데이터 요청 중 → 스피너/스켈레톤
3. **데이터 있음**: 정상 로드 → 콘텐츠 표시
4. **빈 상태**: 조건 맞는 데이터 없음 → EmptyState 안내
5. **에러**: 요청 실패 → 인라인 에러 + 재시도

### 토큰·스타일
- 색상: `var(--color-*)`, `var(--krds-*)`, `var(--narae-*)`만 사용.
- 간격·타이포: CSS 변수 또는 Tailwind 유틸리티. 하드코딩 px/색상코드 금지.
- krds-react 컴포넌트는 래핑해서 사용, 직접 스타일 override 금지.

### 추측 금지
- 명세에 없는 값·동작·데이터는 `[CONFIRM: 구체적 질문]`으로 남기고 사용자에게 묻는다.
- `// TODO: 기획확인` 대신 `[CONFIRM]` 형식을 사용한다.
